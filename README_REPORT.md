# Рабочий журнал разработки MVP для НИР

Документ фиксирует ход разработки проекта «Прототип ETL-конвейера для автоматического извлечения и обогащения метаданных учебных материалов PDF/DOCX/HTML». Его можно использовать как основу разделов «Ход выполнения работы», «Обоснование проектных решений» и «Экспериментальная часть».

## Этап. Проектирование архитектуры ETL-конвейера

### Цель этапа

Согласовать MVP с архитектурной схемой `Пользователь -> Input -> Ingest -> Extract -> Transform -> Validate -> Load -> Output`.

### Что было сделано

- Выбрана последовательная ETL-архитектура.
- Этапы схемы отражены в структуре кода: `src/ingest.py`, `src/extract/*`, `src/transform/*`, `src/validate.py`, `src/load.py`.
- Добавлена ссылка на архитектурную схему `reports/assets/etl_architecture.png`.

### Принятые решения

- Последовательная ETL-архитектура выбрана, потому что она проще для локального MVP и хорошо объясняется в отчете.
- Каждый формат обрабатывается отдельным адаптером, потому что PDF, DOCX и HTML имеют разную внутреннюю структуру.
- Реестр файлов и хеширование добавлены для обнаружения дублей и будущей инкрементальной обработки.
- Transform разделен на очистку, тему, сложность, ключевые слова и автора, чтобы каждый подэтап можно было заменить отдельно.
- Validate вынесен отдельно, потому что качество данных должно контролироваться после обогащения.
- JSON выбран как выходной формат, потому что он удобен для последующей загрузки в поиск, аналитику, LMS и RAG.

### Рассмотренные альтернативы

- Airflow не используется в MVP, потому что оркестратор увеличивает сложность и не нужен для проверки основной идеи.
- Spark не используется, потому что тестовый набор небольшой и локальный Python pipeline легче отладить.
- ML/LLM-классификация не включена, потому что сначала нужен воспроизводимый baseline.

### Возникшие трудности

- Исходная архитектурная схема была дана внешней ссылкой, а не файлом в рабочей папке.

### Как трудности были решены

- В проект добавляется локальная схема `etl_architecture.png`, соответствующая требуемой последовательности этапов.

### Результат этапа

Получена структура MVP, напрямую соответствующая архитектурной схеме.

### Артефакты для отчёта

- Таблица соответствия этапов схемы и модулей.
- `reports/assets/etl_architecture.png`.

## Этап 2. Реализация Ingest

### Цель этапа

Найти поддерживаемые документы, собрать реестр файлов и обнаружить дубликаты.

### Что было сделано

- Реализован `src/ingest.py`.
- Использован рекурсивный обход через `pathlib`.
- Реализован SHA256 через `hashlib` в `src/utils/hashing.py`.
- Для каждого файла сохраняются `file_id`, `source_path`, `file_name`, `file_format`, `file_size_bytes`, `file_hash`, `modified_at`, `ingest_status`, `duplicate_of`.

### Принятые решения

- `pathlib` выбран как современный стандартный API для работы с путями.
- `hashlib` выбран, потому что он входит в стандартную библиотеку и надежно считает SHA256.
- Проверка дублей по хешу позволяет не обрабатывать одинаковые файлы повторно.
- Хеширование создает основу для будущей инкрементальной обработки.

### Рассмотренные альтернативы

- Сравнение файлов по имени не выбрано, потому что разные файлы могут называться одинаково, а одинаковые файлы могут иметь разные имена.

### Возникшие трудности

- Нужно было сохранить дубликаты в лог, но не включать их в повторную обработку.

### Как трудности были решены

- Для дублей выставляется `ingest_status = skipped_duplicate`, а `duplicate_of` указывает на первый найденный файл.

### Результат этапа

Ingest формирует воспроизводимый реестр входных файлов.

### Артефакты для отчёта

- Фрагмент `processing_log.csv` с `skipped_duplicate`.

## Этап 3. Реализация Extract для PDF/DOCX/HTML

### Цель этапа

Извлечь текст и базовые метаданные с учетом различий форматов.

### Что было сделано

- Добавлен общий интерфейс `src/extract/base.py`.
- PDF: `src/extract/pdf_extractor.py`, основной инструмент `pdfplumber`, fallback `PyPDF2`.
- DOCX: `src/extract/docx_extractor.py`, извлечение параграфов, таблиц и core properties.
- HTML: `src/extract/html_extractor.py`, BeautifulSoup, удаление `script`, `style`, `nav`, `footer`, извлечение `title` и `h1/h2/h3`.

### Принятые решения

- `pdfplumber` выбран для постраничного извлечения текста из PDF.
- `PyPDF2` используется как fallback для повышения устойчивости.
- `python-docx` выбран как специализированная библиотека для DOCX.
- BeautifulSoup выбран для HTML, потому что он устойчив к неидеальной разметке.

### Рассмотренные альтернативы

- Универсальный текстовый парсер не выбран, потому что форматы имеют разные модели документа.
- OCR не включен в обязательную часть MVP, потому что обработка сканов требует отдельной настройки.

### Возникшие трудности

- PDF может быть сканом и не содержать текстового слоя.
- Встроенные метаданные часто отсутствуют.
- DOCX может иметь нестандартную структуру.
- HTML может содержать технический мусор.

### Как трудности были решены

- Ошибки сохраняются в `errors`, документ не теряется.
- Для PDF добавлен fallback.
- Для HTML удаляются технические теги перед извлечением текста.

### Результат этапа

Поддержаны три входных формата и единый результат extraction.

### Артефакты для отчёта

- Пример записи документа со статусом `success` или `failed`.

## Этап 4. Реализация Transform/Enrich

### Цель этапа

Очистить текст и обогатить документы аналитическими метаданными.

### Что было сделано

- Очистка: `src/transform/cleaner.py`.
- Тема: `src/transform/topic_classifier.py`.
- Сложность: `src/transform/difficulty_classifier.py`.
- Метаданные: `src/transform/metadata_extractor.py`.
- Добавлены эвристики языка, формул, кода, автора и ключевых слов.

### Принятые решения

- Очистка выполняется regex-правилами, потому что для MVP важны простота и воспроизводимость.
- Тема определяется по словарям ключевых слов, потому что rule-based подход интерпретируем.
- Сложность определяется по объему, терминам, формулам, коду и длине предложений.
- Ключевые слова извлекаются через YAKE, а при недоступности YAKE используется частотный fallback.
- Автор определяется сначала из метаданных, затем по паттернам в начале текста.

### Рассмотренные альтернативы

- Сложная ML/LLM-классификация не выбрана, потому что она требует датасета, настройки и усложняет воспроизводимость.
- spaCy не сделан обязательной зависимостью, потому что модели языка увеличивают вес установки.

### Возникшие трудности

- Rule-based классификация может ошибаться на смешанных темах.
- Автор может отсутствовать или быть указан нестандартно.
- YAKE может быть не установлен.

### Как трудности были решены

- Добавлены значения `unknown` и `null`.
- Добавлен fallback для ключевых слов.
- Ошибки не останавливают весь pipeline.

### Результат этапа

Документы получают `language`, `topic`, `difficulty`, `keywords`, `author`, `has_formulas`, `has_code`, `word_count`, `char_count`.

### Артефакты для отчёта

- Пример enriched JSON.
- Обоснование rule-based baseline.

# Итоговые материалы для отчёта

## Что реализовано в MVP

- Локальный консольный ETL-конвейер.
- Обработка PDF/DOCX/HTML.
- Извлечение текста.
- Извлечение и обогащение метаданных.
- Валидация результата.
- Сохранение JSON-датасета.
- Сбор метрик.
- Генерация графиков.
- Логирование ошибок.

## Основные проектные решения

- Модульная ETL-архитектура.
- Локальный Python pipeline вместо Airflow на этапе MVP.
- Отдельные адаптеры под каждый формат.
- Rule-based topic classification.
- Эвристическое определение difficulty.
- Сохранение ошибок вместо удаления проблемных документов.
- JSON как основной формат выгрузки.
- Pydantic как инструмент контроля качества выходной схемы.

## Основные трудности

- Возможные PDF без текстового слоя.
- Неполные встроенные метаданные.
- Нестандартная структура DOCX.
- HTML с техническим мусором.
- Отсутствие зависимостей в окружении до установки.
- Ограниченная точность rule-based классификации.

## Ограничения MVP

- Нет OCR для сканированных PDF.
- Topic и difficulty определяются эвристически.
- Нет промышленной оркестрации через Airflow.
- Нет распределенной обработки через Spark.
- Нет загрузки в Iceberg/ClickHouse.
- Нет интеграции с LMS.
- Качество метаданных зависит от качества исходных документов.

## Возможности дальнейшего развития

- Подключение OCR.
- Переход к Airflow DAG.
- Масштабирование через Spark.
- Хранение в Iceberg.
- Аналитический слой ClickHouse/Trino.
- Индексация в Elasticsearch/Typesense.

## Этап 11. Пилотная проверка MVP на тестовом наборе PDF/DOCX/HTML

### Цель этапа

Проверить работоспособность конвейера на разных форматах учебных материалов и подготовить артефакты, пригодные для отчёта по НИР.

### Состав тестового набора

Для проверки подготовлена структура `data/real_input/`:

- `pdf/` — расширенный набор реальных PDF с текстовым слоем;
- `docx/` — расширенный набор реальных DOCX и синтетические DOCX-примеры;
- `html/` — сохранённая HTML-страница и 20 страниц, собранных с NEERC wiki;
- `bad_files/` — проблемный файл для проверки устойчивости;
- `duplicates/` — дубликат HTML-файла для проверки `skipped_duplicate`.

Фактический текущий набор содержит 66 поддерживаемых файлов: 18 PDF с учётом плохого PDF и одного PDF-дубликата, 26 DOCX и 22 HTML с учётом HTML-дубликата. Набор является пилотным; целевой объём около 60 документов на текущем этапе достигнут. Расширение до 100+ документов запланировано как следующий этап исследования для получения более устойчивых статистических оценок.

Часть DOCX сформирована синтетически. Это сделано не для завышения результата, а для гарантированной проверки типовых элементов DOCX-адаптера: заголовков, параграфов, таблиц, автора и core properties.

### Анализ результатов

Проверка подтвердила, что конвейер обрабатывает PDF, DOCX и HTML, формирует JSON-датасет, CSV-лог, файл ошибок, метрики и PNG-графики. Для реального PDF использован `pdfplumber`, `page_count` определён как 22, embedded metadata частично заполнены. Для DOCX использован `python-docx`, извлекаются параграфы, таблицы, заголовки и core properties. Для HTML использован BeautifulSoup, извлекаются title, h1/h2/h3 и meta-теги.

Проверка дубликатов выполнена через SHA256. Файлы `Лекция №7 new new.pdf` и `boolean_function_duplicate.html` получили статус `skipped_duplicate`, а в `processing_log_real.csv` поле `duplicate_of` указывает на первый найденный документ.

Проверка плохого PDF подтвердила устойчивость: `pdfplumber` вернул ошибку, был предпринят fallback на `PyPDF2`, после чего документ сохранён со статусом `failed` и списком ошибок. Конвейер продолжил обработку остальных файлов.

Metadata completeness теперь интерпретируется осторожнее. Универсальная полнота равна 0.888672; основной провал связан с полем `author`, которое часто отсутствует в реальных учебных материалах. Форматно-зависимая полнота равна 0.921875; `page_count` применяется только к PDF, поэтому HTML/DOCX не штрафуются за неприменимое поле.

### Ограничения

- Набор превышает 60 файлов, но всё ещё меньше 100 документов и рассматривается как пилотный.
- DOCX-поднабор расширен до 26 файлов; часть документов синтетическая.
- OCR пока не реализован, поэтому сканированные PDF без текстового слоя будут попадать в ошибки.
- Классификация темы и сложности основана на эвристиках и не является ML-моделью.
- Автоматический сбор NEERC HTML работает, но качество ссылок и тематику набора нужно дополнительно курировать.
